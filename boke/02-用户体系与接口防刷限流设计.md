#  趣享社用户体系与接口防刷限流设计深度解析

## 一、项目核心概述

​	用户体系是任何社区平台的基石。 趣享社项目的用户模块不仅仅实现了注册登录，更在安全防护上做了多层加固：BCrypt 不可逆加密防止密码泄露，JWT 双令牌机制在安全性与用户体验间取得平衡，双层限流（用户级 + IP 级）与双算法（滑动窗口 + 固定窗口）组合防止恶意刷接口。同时，Spring Security + JwtAuthenticationFilter 的无状态认证模式让服务横向扩展时无需处理 Session 共享问题。本文将对这套体系的完整实现进行逐层拆解。

## 四、核心方案落地实现

### 4.1 BCrypt 密码加密（PasswordUtil）

密码存储采用 Spring Security 内置的 BCrypt 算法，内置随机盐值，相同的明文每次加密结果不同：

```java
public class PasswordUtil {
    private static final BCryptPasswordEncoder encoder = new BCryptPasswordEncoder();

    public static String encode(String rawPassword) {
        return encoder.encode(rawPassword);
    }

    public static boolean verify(String rawPassword, String encodedPassword) {
        if (rawPassword == null || encodedPassword == null) {
            return false;
        }
        return encoder.matches(rawPassword, encodedPassword);
    }
}
```

BCrypt 的核心优势：
- **内置盐值**：加密结果中包含随机盐（前 22 位），无需额外的盐值存储字段
- **可调节计算强度**：默认 `strength=10`，计算一次约 100ms，暴力破解成本极高
- **防彩虹表攻击**：即使两用户密码相同，密文也完全不同

### 4.2 JWT 双令牌方案（AuthServiceImpl.generateLoginVO）

双令牌机制的核心代码：

```java
private LoginVO generateLoginVO(User user, String ipAddress) {
    // 生成 accessToken（15分钟有效期）
    Map<String, Object> accessClaims = new HashMap<>();
    accessClaims.put("sub", user.getId().toString());
    accessClaims.put("username", user.getUsername());
    accessClaims.put("type", "access");
    accessClaims.put("role", user.getRole() != null ? user.getRole() : User.ROLE_USER);
    String accessToken = jwtUtil.createToken(accessClaims, accessTokenValidity * 1000L);

    // 生成 refreshToken（7天有效期）
    Map<String, Object> refreshClaims = new HashMap<>();
    refreshClaims.put("sub", user.getId().toString());
    refreshClaims.put("type", "refresh");
    String refreshToken = jwtUtil.createToken(refreshClaims, refreshTokenValidity * 1000L);

    // 存储 refreshToken 到 Redis（7天过期）
    redisTokenService.storeRefreshToken(user.getId(), refreshToken, refreshTokenValidity);

    LoginVO vo = new LoginVO();
    vo.setAccessToken(accessToken);
    vo.setRefreshToken(refreshToken);
    vo.setTokenType("Bearer");
    vo.setExpiresIn(accessTokenValidity);
    vo.setUser(userService.convertToVO(user));
    return vo;
}
```

双令牌设计要点：
- **accessToken**：JWT 格式，15 分钟有效期，存储在前端内存/本地存储，每次请求携带在 `Authorization: Bearer xxx` 请求头
- **refreshToken**：JWT 格式，7 天有效期，同时存储在 Redis 中（支持服务端主动失效）。accessToken 过期后，前端用 refreshToken 调用 `/auth/refresh` 接口换取新的双令牌
- **Token 类型校验**：refresh 接口会校验 Claims 中的 `type` 字段必须为 `"refresh"`，防止用 accessToken 冒充 refreshToken

refresh 接口的关键校验逻辑：

```java
public LoginVO refreshToken(String refreshToken, String ipAddress) {
    Map<String, Object> claims = jwtUtil.validateAndGetClaimsFromToken(refreshToken);
    if (claims == null) {
        throw new BusinessException(1003001, "refreshToken无效");
    }
    // 防止 accessToken 被误用为 refreshToken
    String tokenType = (String) claims.get("type");
    if (!"refresh".equals(tokenType)) {
        throw new BusinessException(1003001, "refreshToken无效");
    }
    Long userId = Long.valueOf((String) claims.get("sub"));
    // 校验 Redis 中该用户的 refreshToken 是否与当前一致
    if (!redisTokenService.validateRefreshToken(userId, refreshToken)) {
        throw new BusinessException(1003001, "refreshToken已失效，请重新登录");
    }
    // 生成新的双令牌
    return generateLoginVO(user, ipAddress);
}
```

### 4.3 JwtAuthenticationFilter 无状态认证

```java
import io.jsonwebtoken.Claims;
import jakarta.servlet.FilterChain;
import jakarta.servlet.ServletException;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.security.authentication.UsernamePasswordAuthenticationToken;
import org.springframework.security.core.authority.SimpleGrantedAuthority;
import org.springframework.security.core.context.SecurityContextHolder;
import org.springframework.stereotype.Component;
import org.springframework.web.filter.OncePerRequestFilter;

import java.util.Collections;
import java.util.Date;
import java.util.List;

/**
 * JWT 认证过滤器
 * 作用：对所有请求进行 Token 校验，解析用户信息并放入 SpringSecurity 上下文
 * 执行时机：每个请求进入 Controller 之前执行一次
 * 继承 OncePerRequestFilter：保证单次请求只执行一次过滤
 *
 * @author 小蓝书项目
 */
@Slf4j
@Component
@RequiredArgsConstructor
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    /**
     * JWT 工具类：用于解析 Token、获取载荷信息
     */
    private final JwtUtil jwtUtil;

    /**
     * 核心过滤方法：请求统一入口
     *
     * @param request     请求对象
     * @param response    响应对象
     * @param filterChain 过滤器链
     */
    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain) {

        // 1. 从请求头中获取 Token
        String token = getTokenFromRequest(request);

        // 2. Token 不为空才进行解析
        if (token != null) {
            // 3. 解析 Token 获取载荷（包含用户ID、用户名、角色、过期时间）
            Claims claims = jwtUtil.getClaimsFromToken(token);

            // 4. 校验：载荷不为空 + Token 未过期
            if (claims != null && !claims.getExpiration().before(new Date())) {

                // 5. 从 Token 中解析用户基础信息
                Long userId = Long.parseLong(claims.getSubject());
                String username = (String) claims.get("username");
                String role = (String) claims.getOrDefault("role", "USER");

                // 6. 封装用户角色权限（SpringSecurity 需要权限集合）
                List<SimpleGrantedAuthority> authorities =
                        Collections.singletonList(new SimpleGrantedAuthority("ROLE_" + role));

                // 7. 构建 SpringSecurity 认证对象（存入用户ID、权限）
                UsernamePasswordAuthenticationToken authentication =
                        new UsernamePasswordAuthenticationToken(userId, null, authorities);

                // 8. 将用户信息存入 Request 域，方便后续 Controller 直接获取
                request.setAttribute("userId", userId);
                request.setAttribute("username", username);
                request.setAttribute("role", role);

                // 9. 将认证信息存入 Security 上下文，全局可用
                SecurityContextHolder.getContext().setAuthentication(authentication);
            }
        }

        // 10. 无论是否认证成功，都放行请求
        // 认证失败不拦截，由后续 @PreAuthorize 等权限注解控制
        filterChain.doFilter(request, response);
    }

    /**
     * 从请求头获取 JWT Token
     * 格式：Bearer xxxxxxxx
     *
     * @param request 请求
     * @return token 字符串
     */
    private String getTokenFromRequest(HttpServletRequest request) {
        String bearerToken = request.getHeader("Authorization");
        // 判断请求头是否以 "Bearer " 开头
        if (bearerToken != null && bearerToken.startsWith("Bearer ")) {
            // 截取 "Bearer " 之后的真实 Token
            return bearerToken.substring(7);
        }
        return null;
    }
}
```

​	值得注意的设计：**即使 JWT 解析失败，filter 也会放行请求**。这是因为项目中 `/auth/**` 路径是公开的，如果 filter 直接拦截未认证请求，这些公开接口将无法访问。权限控制完全交给 Spring Security 的 `authorizeHttpRequests` 配置和 Controller 层的 `@PreAuthorize` 注解。

### 4.4 @RateLimit 自定义注解

```java
@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface RateLimit {
    String key() default "";
    int maxRequests() default 10;
    int windowSeconds() default 60;
    LimitType type() default LimitType.FIXED_WINDOW;
    String message() default "请求过于频繁，请稍后再试";

    enum LimitType {
        FIXED_WINDOW,    // 固定窗口，性能高，边界处可能突增
        SLIDING_WINDOW   // 滑动窗口，流量更平滑，成本略高
    }
}
```

### 4.5 滑动窗口限流器（Redis ZSet + Lua 脚本）

​	滑动窗口比固定窗口更平滑，因为它在任意时刻都统计的是"过去 N 秒"内的请求数，不会出现窗口边界突增的"毛刺"问题：

```java
import org.springframework.data.redis.core.RedisTemplate;
import org.springframework.data.redis.core.script.RedisScript;
import org.springframework.stereotype.Component;
import java.util.List;

/**
 * 基于 Redis + Lua 实现的**滑动窗口限流**工具
 * 用途：接口防刷、防止恶意请求、保护核心接口（发布笔记、审核、Feed流）
 * 特点：
 * 1. 精确滑动窗口，无固定窗口边界毛刺
 * 2. Lua 脚本保证原子性，高并发下绝对安全
 * 3. 基于 ZSet 实现，自动清理过期数据
 *
 * @author 趣享社项目
 */
@Component
public class SlidingWindowRateLimiter {

    /**
     * Redis Lua 脚本：滑动窗口限流核心逻辑（原子执行）
     * 逻辑说明：
     * 1. 计算窗口起始时间 = 当前时间 - 窗口大小
     * 2. 移除窗口外的所有请求记录（ZREMRANGEBYSCORE）
     * 3. 统计窗口内剩余请求数（ZCARD）
     * 4. 未超限：添加当前请求时间戳 + 随机值（避免重复），设置过期时间
     * 5. 已超限：返回 0 拒绝
     *
     * 返回：1=允许  0=拒绝
     */
    private static final String SLIDING_WINDOW_SCRIPT =
        "local key = KEYS[1] " +
        "local now = tonumber(ARGV[1]) " +
        "local window = tonumber(ARGV[2]) " +
        "local limit = tonumber(ARGV[3]) " +
        "local windowStart = now - window " +
        "redis.call('ZREMRANGEBYSCORE', key, 0, windowStart) " +
        "local count = redis.call('ZCARD', key) " +
        "if count < limit then " +
        "  redis.call('ZADD', key, now, now .. ':' .. math.random()) " +
        "  redis.call('EXPIRE', key, math.ceil(window / 1000) + 1) " +
        "  return 1 " +
        "end " +
        "return 0";

    private final RedisTemplate<String, Object> redisTemplate;

    public SlidingWindowRateLimiter(RedisTemplate<String, Object> redisTemplate) {
        this.redisTemplate = redisTemplate;
    }

    /**
     * 尝试获取限流许可
     *
     * @param key           限流key（如：userid、ip、接口名）
     * @param maxRequests   窗口内最大允许请求数
     * @param windowSeconds 滑动窗口大小（秒）
     * @return true=允许通行  false=被限流
     */
    public boolean tryAcquire(String key, int maxRequests, int windowSeconds) {
        // 组装Redis完整Key
        String fullKey = "rate_limit:sliding:" + key;

        // 当前时间戳（毫秒）
        long nowMillis = System.currentTimeMillis();

        // 窗口大小转为毫秒
        long windowMillis = windowSeconds * 1000L;

        // 执行Lua脚本，保证原子性
        Long result = redisTemplate.execute(
            RedisScript.of(SLIDING_WINDOW_SCRIPT, Long.class),
            List.of(fullKey),
            String.valueOf(nowMillis),
            String.valueOf(windowMillis),
            String.valueOf(maxRequests)
        );

        // 返回是否允许访问
        return result != null && result == 1;
    }
}
```

​	Lua 脚本确保"删除过期记录 + 计数 + 判断 + 添加"四个步骤在 Redis 服务端**原子执行**，不存在并发竞态。相比固定窗口（在窗口切换的边界可能瞬间放行两倍流量），滑动窗口的流量控制更加精准。

### 4.6 双层限流 AOP（RateLimitAspect）

```java
@Aspect
@Component
@RequiredArgsConstructor
public class RateLimitAspect {

    @Around("@annotation(rateLimit)")
    public Object around(ProceedingJoinPoint joinPoint, RateLimit rateLimit) throws Throwable {
        HttpServletRequest request = getHttpRequest();
        if (request == null) {
            return joinPoint.proceed(); // 非HTTP请求直接放行
        }
        String clientIp = getClientIp(request);
        String rateLimitKey = buildKey(rateLimit.key(), joinPoint, clientIp);

        // 第一层：用户级限流
        boolean userAllowed = checkRateLimit(rateLimitKey, rateLimit);
        // 第二层：IP级兜底限流（硬编码30次/分钟）
        boolean ipAllowed = fixedWindowRateLimiter.tryAcquire("rate:ip:" + clientIp, 30, 60);

        if (!userAllowed) {
            throw new BusinessException(429, rateLimit.message());
        }
        if (!ipAllowed) {
            throw new BusinessException(429, "操作过于频繁，请稍后重试");
        }
        return joinPoint.proceed();
    }
}
```

​	双层限流的精妙之处：**用户级限流**可以通过注册更多账号绕过（僵尸账号攻击），但 **IP 级限流**是硬编码 30 次/分钟的最后防线。即使攻击者注册 100 个账号并发请求，每个 IP 的请求总量仍被限制。

### 4.7 路径级限流拦截器（RateLimitInterceptor）

与注解模式互补，拦截器用于对**路径前缀**进行粗粒度限流，配置集中在 Map 中，方便统一管理：

```java
@Component
@RequiredArgsConstructor
public class RateLimitInterceptor implements HandlerInterceptor {
    private static final Map<String, RateLimitConfig> LIMIT_CONFIG = new ConcurrentHashMap<>();

    static {
        LIMIT_CONFIG.put("/api/auth/login",       new RateLimitConfig(10, 60));  // 登录
        LIMIT_CONFIG.put("/api/auth/register",    new RateLimitConfig(5, 60));   // 注册
        LIMIT_CONFIG.put("/api/auth/sendCode",    new RateLimitConfig(1, 60));   // 验证码
        LIMIT_CONFIG.put("/api/note",             new RateLimitConfig(10, 60));  // 发笔记
        LIMIT_CONFIG.put("/api/note/*/like",      new RateLimitConfig(30, 60));  // 点赞
        LIMIT_CONFIG.put("/api/note/*/favorite",  new RateLimitConfig(30, 60));  // 收藏
    }

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
        String uri = request.getRequestURI();
        String userId = getUserId(request); // 已登录用userId，未登录用IP
        RateLimitConfig config = findMatchingConfig(uri);
        if (config == null) return true;

        String rateLimitKey = uri + ":" + userId;
        boolean allowed = rateLimiter.tryAcquire(rateLimitKey, config.maxRequests, config.windowSeconds);
        if (!allowed) {
            response.setStatus(429);
            response.getWriter().write("{\"code\":429,\"message\":\"请求过于频繁，请稍后再试\"}");
            return false;
        }
        return true;
    }
}
```

## 五、多方案横向对比

### 5.1 认证方案：JWT无状态 vs Session有状态

| 维度 | JWT 双令牌 | Session + Cookie |
|---|---|---|
| 水平扩展 | 无需共享Session，天然支持 | 需要Redis共享Session或IP哈希 |
| 安全性 | Token泄露后可在Redis中标记失效 | Session可服务端主动销毁 |
| 移动端适配 | 原生支持，放Header即可 | Cookie机制在App中不友好 |
| 存储开销 | 客户端存储，服务端仅存refreshToken | 服务端需存储全部Session |

### 5.2 密码加密：BCrypt vs MD5 vs SHA-256

| 维度 | BCrypt | MD5/SHA-256 |
|---|---|---|
| 盐值 | 内置随机盐，自动管理 | 需要额外设计盐值机制 |
| 计算速度 | 慢（约100ms），防暴力破解 | 快（微秒级），易被暴力破解 |
| 防彩虹表 | 天然防 | 需要额外加盐 |
| 计算强度 | 可调节 | 固定 |

## 六、项目选型原因

​	选择 JWT 双令牌而非 Session 的核心原因是**服务水平扩展的零成本**。在 Session 方案下，后端服务扩容需要考虑 Session 同步问题（Redis 集中存储或 IP Hash 亲和性），而 JWT 的 Token 自包含特性让任何服务实例都能独立验证请求身份。双令牌设计则解决了纯 JWT 方案最大的痛点——token 泄露后无法主动失效：我们将 refreshToken 存储到 Redis，登出或修改密码时只需删除 Redis 中的 refreshToken，即使 accessToken 还有几分钟有效期，也无法再续期。

​	选择 BCrypt 的理由简洁而坚定：**它是目前业界公认最安全的密码存储方案之一**，Spring Security 内置了对它的完美支持。相比 MD5/SHA-256 需要自行设计盐值逻辑且计算过快容易暴力破解，BCrypt 的慢哈希特性天然适合密码存储场景。

## 七、踩坑记录与问题解决

**踩坑 1：限流 Lua 脚本中的 ZADD member 冲突**

​	初版滑动窗口脚本使用 `now`（时间戳）直接作为 ZSet 的 member，结果在同一毫秒内两个并发请求写入时，第二个请求的 `ZADD` 会覆盖第一个的记录，导致计数低于实际请求数，漏过限流。

**解决方案**：在 member 后拼接随机数：
```lua
redis.call('ZADD', key, now, now .. ':' .. math.random())
```

**踩坑 2：JWT Claims 中 userId 类型不一致**

​	生成 JWT 时 `user.getId()` 返回 Long，存入 `Claims` 时自动转为 Integer（值小于 int 最大值时），但 `getUserIdFromToken` 中用 `Long.parseLong()` 解析。当 userId 超过 Integer.MAX_VALUE 后不会出问题，但在某些序列化场景下可能丢失精度。统一使用 `String.valueOf()` 存入再解析回来，避免了类型转换陷阱。

## 八、性能优化与高并发设计

​	**1. Lua 脚本原子化**：限流判断的所有步骤（清理过期数据、统计、判断、添加记录）在一个 Lua 脚本中完成，Redis 单线程执行 Lua 脚本保证原子性，避免了多命令的网络往返和竞态条件。

​	**2. 限流异常兜底放行**：限流组件出现 Redis 连接异常时，两个限流器都设计了 `catch(Exception) return true` 的兜底逻辑，避免限流组件故障导致所有请求被拦截。

​	**3. 分层限流降低 Redis 压力**：路径级拦截器只对约 10 个高频路径进行限流，注解级 AOP 只对标注的方法进行限流。绝大多数请求不需要额外 Redis 调用，整体性能开销极低。

​	**4. refreshToken 批量失效**：用户修改密码时调用 `redisTokenService.removeRefreshToken(userId)` 仅一次 Redis DEL 操作即可让该用户所有设备的 refreshToken 同时失效，效率极高。

## 九、总结与后续迭代方向

​	趣享社的用户体系以 JWT 双令牌 + BCrypt + 双层限流构建了坚实的安全基础。访问令牌的短有效期降低了泄露风险，刷新令牌的服务端存储让令牌可控。滑动窗口 + 固定窗口的双算法组合，结合用户级 + IP 级的双层策略，有效防止了接口被刷。

**后续迭代方向**：

1. **多因素认证**：增加手机验证码、TOTP 两步验证作为可选安全等级
2. **设备指纹**：结合 User-Agent、设备 ID 做异常登录检测，异地登录时触发二次验证
3. **Sentinel 集成**：将自研限流替换为阿里 Sentinel，获得更完善的流控面板和热点参数限流
4. **OAuth2 扩展**：支持 GitHub、Google 等第三方账号绑定与登录
